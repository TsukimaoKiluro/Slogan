## 一、核心设计思想：约定优于配置
Spring Boot 的底层哲学是 **Convention Over Configuration**。框架通过提供一套**合理的默认值**，极大降低了项目初始化的心智负担。

- **核心表现**：无需编写 `web.xml`、无需配置外部 Tomcat、数据源自动识别。
- **覆盖原则**：开发者只需在 `application.yml/properties` 中提供**显式配置**，即可覆盖默认行为。
- **价值**：将开发重心从**环境搭建与样板代码**转移至**业务逻辑实现**，实现开箱即用。

## 二、Starter 依赖原理
Starter 并非实际的功能代码库，而是一组**依赖描述符（POM Descriptor）**，用于聚合特定场景所需的第三方库。

### 1. 命名规范
官方 Starter 统一遵循 `spring-boot-starter-*` 格式（如 `spring-boot-starter-web`、`spring-boot-starter-data-jpa`）。第三方扩展推荐 `xxx-spring-boot-starter`。

### 2. 版本仲裁（BOM 机制）
通过父工程 `spring-boot-dependencies` 提供**统一版本管理**，避免依赖冲突（Dependency Hell）。
```xml
<!-- 父工程隐式引入版本仲裁 -->
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.2.0</version>
</parent>

<!-- 实际引入的 Starter 无需指定 version -->
<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
</dependencies>
```
**💡 核心机制**：`<dependencyManagement>` 锁定版本 + 传递依赖自动引入。

## 三、自动配置机制（核心）
自动配置是 Spring Boot 的灵魂。框架在启动时扫描 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件，加载所有候选配置类，并通过 **条件注解（Conditional Annotations）** 进行动态装配。

### 1. 条件装配数学模型
自动配置类的注册过程可抽象为以下逻辑判断：
$$
\text{RegisterBean} = (\text{ClassExists} \in \text{ClassPath}) \land (\text{PropertyMatch} \in \text{Env}) \land (\text{BeanMissing} \in \text{Context})
$$

### 2. 核心条件注解工作逻辑
| 注解 | 触发时机 | 典型应用场景 |
|:---|:---|:---|
| `@ConditionalOnClass` | **类路径存在指定 Class** 时生效 | 检测是否引入了 `DataSource` 驱动类 |
| `@ConditionalOnMissingBean` | **容器中不存在指定类型/名称的 Bean** 时生效 | 用户未自定义 `DataSource` 时提供默认实现 |
| `@ConditionalOnProperty` | **配置文件匹配指定 key-value** 时生效 | `spring.datasource.enabled=true` |

### 3. 源码级执行流程
```java
@Configuration(proxyBeanMethods = false)
@ConditionalOnClass({ DataSource.class, EmbeddedDatabaseType.class })
@EnableConfigurationProperties(DataSourceProperties.class)
@Import({ DataSourcePoolMetadataProvidersConfiguration.class, DataSourceInitializationConfiguration.class })
public class DataSourceAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean // 核心：用户未自定义才装配
    public DataSource dataSource() {
        // 根据 HikariCP/Tomcat/Druid 的 classpath 优先级创建默认数据源
        return createDataSource(properties);
    }
}
```
**⚠️ 优先级规则**：开发者自定义 Bean > `@ConditionalOnMissingBean` 生成的 Bean。

## 四、内嵌服务器优势
Spring Boot 默认将 **Tomcat / Jetty / Undertow** 直接打包进可执行 Jar，彻底改变传统部署模式。

1. **独立运行**：`java -jar app.jar` 即可启动，无需外部应用服务器。
2. **环境一致性**：开发、测试、生产使用完全相同的 Servlet 容器版本，消除“在我机器上能跑”的问题。
3. **云原生友好**：轻量级镜像构建、支持水平扩缩容、完美契合 Kubernetes Pod 生命周期。
4. **按需切换**：通过排除依赖可无缝替换内嵌服务器（如排除 Tomcat 引入 Undertow）。
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <exclusions>
        <exclusion>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-tomcat</artifactId>
        </exclusion>
    </exclusions>
</dependency>
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-undertow</artifactId>
</dependency>
```

## 五、`@SpringBootApplication` 三合一注解拆解
该注解是 Spring Boot 应用的**入口标识**，本质是一个组合注解：
```java
@SpringBootConfiguration  // 1. 标记为配置类，底层继承 @Configuration
@EnableAutoConfiguration  // 2. 开启自动配置机制，导入 AutoConfigurationImportSelector
@ComponentScan            // 3. 组件扫描，默认扫描主类所在包及子包
public @interface SpringBootApplication {
    // 属性可覆盖：exclude = {DataSourceAutoConfiguration.class} 等
}
```
- **`@SpringBootConfiguration`**：允许使用 `@Bean` 声明自定义组件。
- **`@EnableAutoConfiguration`**：通过 `ImportSelector` 延迟加载并过滤自动配置类。
- **`@ComponentScan`**：自动注册 `@Controller`、`@Service`、`@Repository` 等注解类。

## 六、启动条件思考题与解析提示

### 🤔 思考题 1：冲突覆盖问题
> 项目中同时引入了 `spring-boot-starter-web`，且开发者在配置类中手动 `@Bean` 定义了一个 `TomcatServletWebServerFactory`。启动后使用的是哪个服务器配置？为什么？

💡 **解析提示**：
- 定位 `ServletWebServerFactoryAutoConfiguration`。
- 查找其工厂 Bean 声明处的条件注解。
- 结合 **`@ConditionalOnMissingBean`** 的优先级机制分析。

### 🤔 思考题 2：条件阻断排查
> 移除 `pom.xml` 中的 `spring-boot-starter-web` 依赖后，应用启动不再监听 8080 端口，且 `DispatcherServlet` 未注册。请描述框架内部的拦截链路。

💡 **解析提示**：
- 追踪 `@ConditionalOnClass` 对 `DispatcherServlet.class` 的检测逻辑。
- 理解自动配置类未满足条件时的 **短路（Short-circuit）** 行为。
- 思考 `spring.factories` / `AutoConfiguration.imports` 的加载与过滤时机。

## 七、总结与课后练习

### 📌 核心总结
1. **约定优于配置** 是 Spring Boot 降低学习成本与提升开发效率的基石。
2. **Starter** 通过传递依赖与 BOM 版本仲裁，实现场景化一键集成。
3. **自动配置** 依赖 `@Conditional` 家族进行动态装配，`@ConditionalOnClass` 决定“能不能配”，`@ConditionalOnMissingBean` 决定“要不要配”。
4. **内嵌服务器** 实现了应用自包含部署，是微服务与云原生架构的标准范式。
5. `@SpringBootApplication` 是配置类、自动装配开关与包扫描的**统一入口**。

### 📝 课后练习
1. **实战调试**：在启动参数中添加 `--debug` 或配置 `logging.level.org.springframework.boot.autoconfigure=DEBUG`，观察控制台打印的 `CONDITIONS EVALUATION REPORT`，找出 3 个因条件不满足而被跳过的自动配置类，并分析原因。
2. **自定义 Starter**：编写一个 `mybatis-plus-spring-boot-starter` 简化版，要求包含：自动配置类、`@ConditionalOnClass(MyBatisPlusAutoConfiguration.class)` 拦截、`application.yml` 属性绑定，并打包供本地项目引用。
3. **原理问答**：为什么 `@EnableAutoConfiguration` 要放在 `@ComponentScan` 之前执行？如果手动调整注解顺序（假设可调整），会对 Bean 的注册时机产生什么影响？
