## 第1页：Spring Boot 技术导论
- 核心定位：基于 Spring 生态的“约定优于配置”快速开发框架，非替代而是架构增强
- 演进目标：降低企业级 Java 项目初始化门槛，统一微服务技术栈标准
- 适用场景：云原生应用、RESTful API 服务、快速原型验证、传统 Spring 项目平滑迁移
- 【示意图占位】Spring 生态演进时间轴 / 传统架构 vs Boot 架构对比简图

## 第2页：传统 Spring vs Spring Boot 核心对比
- 配置复杂度：XML/显式 JavaConfig 手动装配 vs 零配置/约定默认值自动推导
- 依赖治理：人工排查版本冲突与传递依赖 vs BOM 统一管控 Starter 依赖树
- 部署范式：WAR 打包部署至外部容器 vs Fat Jar 独立进程一键启动
- 运维体验：日志/监控需手动集成 vs Actuator/DevTools 开箱即用
- 【示意图占位】配置代码量对比柱状图 / 部署流程差异流程图

## 第3页：核心特性一：Starter 依赖管理机制
- 聚合设计：按业务场景预组装依赖（如 `spring-boot-starter-web` 集成 MVC/Tomcat/Jackson）
- 版本仲裁：`spring-boot-dependencies` 集中声明版本，彻底解决“Jar Hell”问题
- 扩展策略：支持在 `pom.xml`/`build.gradle` 中覆盖特定版本，兼容企业定制化需求
- 【示意图占位】Starter 依赖传递树状图 / 依赖仲裁优先级示意图

## 第4页：核心特性二：内嵌服务器与部署架构
- 容器内嵌：Tomcat/Jetty/Undertow 以 Jar 形式嵌入，彻底解耦外部应用服务器
- 启动链路：`ServletWebServerApplicationContext` 动态初始化并绑定网络端口
- 进程模型：从“WAR 部署至容器”转向“独立可执行进程”，天然契合 Docker/K8s 编排
- 【示意图占位】内嵌容器启动生命周期图 / 传统 WAR vs Fat Jar 目录结构对比

## 第5页：核心特性三：自动配置（Auto-Configuration）
- 设计哲学：基于 Classpath 依赖、IoC 容器状态及外部配置，动态决策 Bean 装配
- 触发入口：读取 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`
- 条件控制：依赖 `@Conditional` 系列注解实现按需加载，避免冗余初始化与 Bean 冲突
- 【示意图占位】自动配置加载时机序列图 / 条件匹配决策树简图

## 第6页：深度拆解：@SpringBootApplication
- `@SpringBootConfiguration`：标识主类为配置类，底层等价于 `@Configuration`
- `@EnableAutoConfiguration`：开启自动配置，核心由 `AutoConfigurationImportSelector` 驱动
- `@ComponentScan`：默认扫描主类所在包及子包，支持 `excludeFilters` 精准控制范围
- 扩展实践：通过 `exclude` 属性禁用冲突自动配置，或结合 `@ComponentScan` 调整扫描边界
- 【示意图占位】@SpringBootApplication 组合注解结构图 / 组件扫描路径示意图

## 第7页：自动配置条件注解工作流（文字版）
- 阶段一：加载候选配置类，解析全量 `@Configuration` 候选集
- 阶段二：执行 `@ConditionalOnClass` / `@ConditionalOnMissingClass` 验证 Classpath 环境
- 阶段三：执行 `@ConditionalOnBean` / `@ConditionalOnMissingBean` 检查 IoC 容器注册状态
- 阶段四：执行 `@ConditionalOnProperty` / `@ConditionalOnResource` 校验配置文件与资源文件，最终实例化目标 Bean
- 【示意图占位】条件匹配流水线逻辑图 / 配置加载与过滤决策流

## 第8页：启动条件思考题与实战引导
- 思考一：Classpath 同时存在多个数据源 Starter 时，Boot 如何依据条件注解排他装配？
- 思考二：如何封装企业级中间件，编写自定义 Starter 并实现自动配置 SPI 接入？
- 引导实践：引入 `spring-boot-starter-actuator`，访问 `/actuator/conditions` 端点可视化装配结果
- 调试技巧：启动参数添加 `--debug` 输出 `CONDITIONS EVALUATION REPORT`，精准定位跳过或匹配项

## 第9页：总结
- 核心价值：以约定替代繁琐配置，以 Starter 统一技术栈，以自动配置赋能敏捷交付
- 关键机制：掌握依赖仲裁、内嵌容器、条件装配三大支柱，是构建云原生微服务的基石
- 学习路径：官方 Reference Docs 精读 → Starter 源码走读 → 自定义 AutoConfiguration 实战 → 深入 Actuator 与 JVM 调优
