## 第1页：传统 Spring vs Spring Boot：配置复杂度对比
- XML/JavaConfig 时代：需手动定义 Bean、配置数据源、MVC 组件及事务管理器，配置量随业务规模线性膨胀
- 依赖管理痛点：显式声明大量第三方库坐标，版本仲裁依赖开发者经验，`ClassNotFoundException` 与依赖冲突频发
- Spring Boot 破局：贯彻“约定优于配置”理念，内置默认配置模板与自动化装配引擎，将初始搭建成本从百行代码压缩至零配置
- [示意图占位：传统Spring配置树状图 vs Spring Boot 扁平化自动装配结构对比]

## 第2页：传统 Spring vs Spring Boot：部署方式演进
- 传统模式：强制打包为 WAR 文件，强依赖外部 Servlet 容器（Tomcat/Jetty），环境耦合度高且运维链路长
- Boot 模式：默认构建为可执行 Fat JAR（`spring-boot-maven-plugin` 打包），通过 `java -jar` 一键拉起，实现“一次构建，处处运行”
- 架构优势：内嵌容器与应用同生命周期，天然适配云原生架构，大幅简化 Docker 镜像构建与 K8s 弹性扩缩容流程
- [示意图占位：外部Tomcat部署链路图 vs 内嵌JAR独立进程启动链路图]

## 第3页：核心特性一：Starter 依赖管理机制
- 聚合坐标设计：将特定功能场景的依赖打包为单一 Starter（如 `spring-boot-starter-web`），屏蔽底层依赖传递复杂性
- 统一版本仲裁：依托 `spring-boot-dependencies` BOM（Bill of Materials）集中管理数百个第三方库版本，彻底解决依赖冲突
- 按需加载原则：Starter 仅提供基础 POM 坐标，实际依赖通过 Maven/Gradle 传递性解析，保持项目依赖树轻量化
- 命名规范约束：官方 Starter 采用 `spring-boot-starter-*` 前缀，社区/第三方采用 `xxx-spring-boot-starter` 后缀，避免命名冲突

## 第4页：核心特性二：内嵌服务器与独立运行
- 容器内置集成：默认打包 Tomcat，通过配置可无缝切换 Jetty 或 Undertow，消除外部中间件安装与调优成本
- 启动引导机制：`SpringApplication.run()` 统一初始化 IoC 容器、加载 Environment 并启动内嵌 Web 服务器监听端口
- 优雅停机保障：应用接收到 `SIGTERM` 信号时自动触发 `DisposableBean` 与 `@PreDestroy`，确保数据库连接池释放与事务回滚

## 第5页：核心特性三：自动配置（Auto-Configuration）原理
- 核心逻辑：基于 Classpath 存在的类库、已定义的 Bean 及环境变量，动态推断并注册所需配置类
- 加载入口：`@EnableAutoConfiguration` 触发 `AutoConfigurationImportSelector` 读取 `META-INF/spring/...AutoConfiguration.imports`
- 优先级覆盖：开发者自定义的 `@Bean` 或 `application.yml` 配置优先级高于自动配置，遵循“用户配置优先”原则
- 调试支持：启动参数添加 `--debug` 或访问 Actuator `/actuator/conditions` 端点，输出详细的自动配置匹配报告

## 第6页：`@SpringBootApplication` 注解深度拆解
- 复合注解本质：等效于 `@SpringBootConfiguration` + `@EnableAutoConfiguration` + `@ComponentScan` 的声明式聚合
- `@SpringBootConfiguration`：标识当前类为根配置类，继承标准 `@Configuration`，允许使用 Java 代码替代 XML 定义 Bean
- `@ComponentScan`：默认扫描主启动类所在包及其所有子包，精准定位 `@Service`/`@Controller` 等组件，避免全量扫描性能损耗
- `@EnableAutoConfiguration`：激活自动配置引擎，是 Spring Boot 实现“开箱即用”的核心开关与装配入口

## 第7页：自动配置条件注解工作流程
- 逻辑起点：容器启动时解析 `@Conditional` 元注解，由 `ConditionEvaluator` 统一拦截并评估装配条件
- 环境快照采集：读取当前 Classpath 类列表、系统属性、`Environment` 配置项及已注册的 BeanDefinitionMap
- 短路匹配机制：依次执行 `matches()` 方法，若任意前置条件不满足（如 `@ConditionalOnMissingBean` 命中），则直接跳过该配置类
- 动态开关控制：结合 `@ConditionalOnProperty(name="xx.enabled", havingValue="true")` 实现运行时特性启停与灰度发布

## 第8页：启动条件思考题与场景引导
- 问题一：移除 `spring-boot-starter-web` 后应用仍能启动，为何无法响应 HTTP 请求？（引导：理解 `SpringApplication` 上下文启动与 `ServletWebServerFactory` 条件装配的差异）
- 问题二：多数据源架构下，`DataSourceAutoConfiguration` 为何自动失效？（引导：分析 `@ConditionalOnMissingBean` 触发机制与手动覆盖的 `@Bean` 优先级）
- 问题三：如何基于 `@ConditionalOnProperty` 实现生产环境日志级别动态降级？（引导：结合配置中心热更新与条件装配实现无停机策略切换）
- 探索路径：开启 `spring.autoconfigure.exclude` 排除特定 Starter，观察启动日志变化，验证条件过滤逻辑

## 第9页：最佳实践与避坑指南
- 包结构规范：主启动类必须置于项目根包，确保 `@ComponentScan` 默认路径覆盖所有业务模块与组件
- 配置分层管理：严格区分 `application.yml`（全局默认）与环境专属文件（`-dev.yml`, `-prod.yml`），利用 Profile 隔离敏感信息
- 依赖精简审计：定期执行 `mvn dependency:tree` 排查冗余传递依赖，避免引入未使用的 Starter 导致内存占用上升
- 可观测性集成：生产环境强制启用 Actuator 健康探针与 Metrics 暴露，结合 Prometheus/Grafana 构建全链路监控体系

## 第10页：总结与进阶路线
- 核心价值：Spring Boot 通过 Starter 聚合、条件装配与内嵌容器，将 Java 企业级开发从“配置驱动”转型为“约定驱动”
- 架构定位：并非替代 Spring Framework，而是提供标准化、生产就绪的 Spring 应用脚手架与自动化编排层
- 技术演进：深入自动配置源码 -> 掌握 Spring Cloud 微服务治理 -> 探索 Spring Boot 3 + GraalVM 原生镜像编译
- 互动环节：开放源码断点调试演示、自定义 Starter 编写实战、容器化部署疑难解答
